# Implementation Plan: 個人用 ToDo アプリ

**Branch**: `001-todo-app` | **Date**: 2026-02-26 | **Spec**: [specs/001-todo-app/spec.md](spec.md)
**Input**: ユーザーからの詳細な実装計画要望（Next.js App Router, Tailwind, Shadcn/ui, ローカル永続化, a11y, 段階的フェーズ）

## 概要

個人が日常のタスクを素早く管理するためのToDoアプリ。Next.js (App Router), Tailwind CSS, shadcn/ui を用いて構築し、オフライン環境下でも動作する完全クライアントサイド（モバイルファースト）な実装を行う。データ永続化は `localStorage` を使用し、a11y（キーボード操作・ラベル・フォーカス可視化）を満たしながら要求されたUI方針（白基調、余白広め）を実現する。

## 技術コンテキスト

**言語/バージョン**: TypeScript / Node.js（Latest LTS）
**主要依存ライブラリ**: Next.js (App Router), React, Tailwind CSS, shadcn/ui, lucide-react, pnpm
**ストレージ**: `localStorage`（クライアントサイドでの永続化）
**ターゲットプラットフォーム**: Webブラウザ（モバイルファースト・レスポンシブデザイン）
**プロジェクト種別**: 単一Webアプリケーション（ロジック・ストレージはフロントエンドのみ）
**パフォーマンス目標**: ロード1秒未満、操作時のUI反映0.15秒以内
**制約**: オフライン完全動作、データ破損時のクラッシュ回避機構
**規模/スコープ**: 最大1000件程度のタスク数

```text
# Webアプリケーション（Next.js App Router）
src/
├── app/
│   ├── layout.tsx       # 全体レイアウト（フォント、a11y初期設定）
│   └── page.tsx         # メイン画面（タスク一覧・入力・フィルタ）
├── components/
│   ├── ui/              # shadcn/ui コンポーネント群
│   └── feature/         # ToDo固有のコンポーネント
├── hooks/
│   └── use-todos.ts     # localStorage とステートを同期・管理するカスタムフック
├── lib/
│   └── storage.ts       # 永続化ロジック (パース、マイグレーション、容量チェック)
└── types/
    └── index.ts         # データモデル定義 (Task, FilterStatus)
```

**構成方針**: Next.js App Routerの標準的な単一プロジェクト構成。ロジックとUIを分離するため、状態管理は `hooks/` に、ストレージI/Oは `lib/` に配置する。

## マイルストーン（フェーズ）一覧 & ステップ

実装を安全かつ段階的に進めるため、以下のフェーズに分割する。

### Phase 1: 基盤構築とデータ層 (Foundation)
**目的**: プロジェクト構造のセットアップと、状態管理・ローカル永続化のロジックを完成させる。

* **Step 1.1: 初期セットアップ**
  * **作業内容**: Next.js, Tailwind, shadcn/ui (ボタン、インプット、チェックボックス、アイコン類) のインストールと `pnpm` 環境構築。
  * **Done条件**: 開発サーバー起動完了、初期画面が表示される。
  * **影響ファイル**: `package.json`, `tailwind.config.ts`, `components/ui/*`
* **Step 1.2: ストレージ管理実装**
  * **作業内容**: `localStorage` からの読み書き処理 `lib/storage.ts` の実装。データ破損対応（try-catchによる空リストフォールバック）、容量上限エラーハンドリング。
  * **Done条件**: 壊れたJSONを localStorage に入れてもパースエラーでクラッシュせず空配列を返す。
  * **影響ファイル**: `lib/storage.ts`, `types/index.ts`
* **Step 1.3: 状態管理用カスタムフック**
  * **作業内容**: `use-todos.ts` を作成し、追加・完了切替・編集・削除・フィルタ機能を持つロジックを実装。マウント時の localStorage 読み込み。
  * **Done条件**: フックを使い、コンソール上で全 CRUD 操作とフィルタリングが正しく処理・保存される。
  * **該当AC**: FR-006, FR-008

### Phase 2: コアUIとアクセシビリティ
**目的**: 仕様に基づいたUIの構築、モバイルファーストかつa11yを満たす画面を作る。

* **Step 2.1: モバイルファーストレイアウト**
  * **作業内容**: `app/page.tsx` において、白基調・余白広め・角丸のメインレイアウトを実装。
  * **Done条件**: スマホ画面サイズで破綻せず、レスポンシブに表示されること。
  * **影響ファイル**: `app/page.tsx`, `app/globals.css`
* **Step 2.2: タスク追加と一覧表示**
  * **作業内容**: `task-input` および `task-item` コンポーネント。インプットフォームの作成とタスクレンダリング。
  * **Done条件**: 空白入力の防止確認、追加後の即時リセット・一覧反映。a11yラベル(aria-label)が設定されていること。
  * **該当AC**: User Story 1 (AC1), FR-001, FR-009, FR-011
* **Step 2.3: タスクの完了切替とフィルタリング**
  * **作業内容**: タスク個別のトグルボタン実装、および画面上部または下部での `filter-bar` (すべて/未完了/完了) の実装。
  * **Done条件**: トグルによる打ち消し線の即時反映、フィルタ切替による表示タスクの正確な絞り込み。
  * **該当AC**: User Story 1 (AC2), User Story 2 (AC1, 2, 3), FR-002, FR-005

### Phase 3: 拡張機能と堅牢性
**目的**: 編集機能、削除時の意図的UI、テストの実装で完成度を高める。

* **Step 3.1: タスクの編集**
  * **作業内容**: ラベルクリックまたは編集ボタンによるインライン入力モードの実装。確定時の保存処理。
  * **Done条件**: 内容の変更が保存され、リロード後も保持される。a11yフォーカスが適切に当たる。
  * **該当AC**: User Story 1 (AC3), FR-003
* **Step 3.2: 削除の誤操作軽減機構**
  * **作業内容**: 削除ボタン（ゴミ箱アイコン）配置と、スワイプ/2段階確認（「本当に削除しますか？」または一定秒数内の取り消し(Undo) トースト表示）。
  * **Done条件**: ユーザーが操作ミスでタスクを削除してしまわないよう、安全なUIフローが機能する。
  * **該当AC**: User Story 1 (AC4), FR-004
* **Step 3.3: フォーカス・キーボード操作対応**
  * **作業内容**: 全体を通じた `focus-visible` のスタイリング確認、Tab/Enter によるフォーカストラップまたは順次移動の最適化。
  * **Done条件**: キーボードのみで追加〜完了〜削除まで完遂できる。フォーカス枠が明確に視認できる。
  * **該当AC**: FR-004, FR-009, FR-010

* **Step 3.4: オフライン起動のサポート (PWA)**
  * **作業内容**: `next-pwa` または組み込みの Service Worker を設定し、静的アセットをキャッシュする。
  * **Done条件**: ネットワークを切断した状態でリロードしても、即座にUIがロードされ操作が可能であること。
  * **該当AC**: FR-007, SC-002

## 4. 動作確認チェックリスト (Manual Verification)

UIとロジック実装完了後、以下の手順で手動テストまたはE2Eテストをパスすること。

- [ ] **追加と復元**: テキストを入力してEnter。直後にリロード(F5)し、入力したタスクおよび未完了状態が維持されているか？
- [ ] **空白回避**: スペースだけで構成されるテキストを入力した際に、追加されず無効化されるか？
- [ ] **完了/未完了切替**: チェックボックスをクリック時、瞬時にテキストへ取り消し線が入るか（状態が更新されるか）？
- [ ] **編集（保存 / キャンセル）**: テキスト部分をダブルクリックまたは「編集」を選択し変更、Enterで反映。空欄にした場合は変更がキャンセル（または削除遷移）されるか？
- [ ] **削除（誤操作対策）**: 削除ボタン押下時、即座に消えるのではなく確認ステップ（ダイアログ/Undoトースト）が提示・機能するか？
- [ ] **フィルタ動作**: 完了済みタスクと未完了タスクを作った上で「未完了」「完了済み」タブを切り替え、正しく該当タスクのみが表示されるか？
- [ ] **クラッシュ回避（データ破損）**: DevTools > Application > LocalStorage に行きデータを `[invalid string}` と書き換えリロード。画面が真っ白にならず、空のタスクリスト（初期状態）として起動するか？
- [ ] **アクセシビリティ（キーボードのみ操作）**: マウスを使わず、Tabキーで順番に要素を移動し、Enterキー等で「追加・編集・完了切替・削除・フィルタ変更」ができるか？またフォーカスされている要素が枠線等で明確か？
- [ ] **極端に長い名前**: 文字数1000文字のタスクを追加した際、他のレイアウトやボタンが画面外に押し出されず、綺麗に折り返されるか（あるいは3点リーダー省略されるか）？

## 5. リスク / 未決事項 (決めるべき順番つき)

以下の事項は実装（Phase 1開始）前または途中に判断を確定させる。

1. **削除処理の具体的なUIパターンの選定**: (最優先)
   - *選択肢A*: 削除ボタン押下時に「確認ダイアログ（Confirmation Modal）」を出す。（実装容易だが手が止まる）
   - *選択肢B*: 削除後、画面下部に「元に戻す（Undo）」のスナックバートーストを数秒出す。（UXが良いが状態管理が複雑）
   - *提案*: **B（Undoトースト）**を選択。shadcn/ui には Toast コンポーネントが存在し、モダンな要件に合致するため。
2. **データの上限制御（エッジケース対応）**:
   - `localStorage` クォータ越えの例外発生時の挙動。厳密なバイト数管理を入れるか、`try-catch` による一律のエラーハンドラーを設けるかに留めるか（後者で十分と想定）。
