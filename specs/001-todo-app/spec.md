# Feature Specification: 個人用 ToDo アプリ

**Feature Branch**: `001-todo-app`  
**Created**: 2026-02-26  
**Status**: Draft  
**Input**: User description: "個人用の ToDo アプリを作りたいです..."

## 目的 (Why)
- やることを素早く追加し、今日やることに集中したい
- タスクが増えても迷わず整理できるようにしたい

## Clarifications
### Session 2026-02-26
- Q: データ保存（ローカル永続化）はLocalStorageとIndexedDBのどちらを用いるか？ → A: LocalStorage (1000件程度のテキストデータ想定のため容量・パフォーマンス要件を十分満たし、実装がシンプルなため)

## User Scenarios & Verification *(mandatory)*

### User Story 1 - タスクの基本操作（追加・完了・編集・削除） (Priority: P1)

ユーザーは、日々のタスクを素早くリストに追加し、完了したものにチェックを入れ、必要に応じて内容を編集したり不要なものを削除したりできる。

**Why this priority**: ToDoアプリの最も核となる機能であり、これがなければアプリとして成立しないため。

**Manual Verification Step**: アプリを起動し、テキスト入力欄からタスクを追加し、リスト上でそのタスクの完了チェックやテキストのインライン編集、さらに削除ボタンによる削除操作が一通り行え、それらが即座に画面に反映されること。

**Acceptance Scenarios**:

1. **Given** 空のToDoリストが表示されている状態、**When** テキスト入力欄にタスク名を入力して追加操作を行う、**Then** 入力したタスクが未完了状態でリストに追加されること。
2. **Given** 未完了のタスクが存在する状態、**When** そのタスクの完了チェックボックスをチェックする、**Then** タスクが完了状態としてマークされ、取り消し線などの視覚的フィードバックがあること。
3. **Given** 既存のタスクが存在する状態、**When** 指定された操作（ダブルクリック等）で編集モードに入り、テキストを変更して保存する、**Then** リスト上のタスク名が即座に更新されること。
4. **Given** 既存のタスクが存在する状態、**When** 削除ボタンを押す、**Then** そのタスクがリストから完全に消去されること。

---

### User Story 2 - タスクのフィルタリング (Priority: P2)

ユーザーは、タスク一覧を「すべて」「未完了」「完了済み」の状態で切り替えて表示できる。

**Why this priority**: タスクが増えてきた際に「今日やるべきこと」に集中しやすくし、整理状態を保つために必要不可欠な機能であるため。

**Manual Verification Step**: 完了済みと未完了のタスクが混在するリストにおいて、各フィルター（すべて/未完了/完了済）を切り替えた際、それぞれの条件に合致するタスクのみが正しく一覧表示されること。

**Acceptance Scenarios**:

1. **Given** 未完了・完了済みのタスクが複数ある状態、**When** 「未完了」フィルターを選択する、**Then** 未完了状態のタスクのみが表示されること。
2. **Given** 未完了・完了済みのタスクが複数ある状態、**When** 「完了済み」フィルターを選択する、**Then** 完了済みのタスクのみが表示されること。
3. **Given** 何らかのフィルターが適用されている状態、**When** 「すべて」フィルターを選択する、**Then** 全てのタスクが状態問わず表示されること。

---

### User Story 3 - データの永続化と安全な復元 (Priority: P1)

ユーザーがアプリをリロードしたり、ブラウザ・アプリを再起動したりしても、タスクのデータと状態が失われず正確に復元される。

**Why this priority**: オフライン前提の要件を満たすためと、ユーザーが入力したデータが失われないという信頼性を担保するため。データ管理の基盤となる。

**Manual Verification Step**: タスクを追加・状態変更した後にページをリロード（再起動）し、内容がそのまま維持されていること。さらにストレージのデータを手動で不正な形式に改変してからリロードしても、クラッシュせず初期状態や安全な形式で立ち上がること。

**Acceptance Scenarios**:

1. **Given** タスクが追加されている状態でアプリを終了する、**When** 次回アプリを起動する、**Then** 終了前と同じタスク一覧と完了状態が復元されること。
2. **Given** アプリがオフライン（ネットワーク未接続）の状態、**When** タスクの追加や編集などの全操作を行う、**Then** 正常に操作が完了し、エラーなくデータがローカルに保存されること。
3. **Given** システムに保存されたローカルデータが意図的に破損している（パース不可な文字列など）状態、**When** アプリを起動する、**Then** アプリがクラッシュすることなく起動し、エラーを安全に捕捉して空のリスト（初期状態）として機能すること。

### Edge Cases / 失敗時の挙動

- **ローカルストレージの上限到達**: LocalStorageの保存領域（約5MB）の容量上限に達した場合、新規追加や更新操作を無効にし、ユーザーに対して「空き容量が不足しています」という警告メッセージを表示してクラッシュを防ぐ。
- **データ形式の後方互換性**: 将来的にデータ構造の変更が発生した場合、旧バージョンのデータ形式を透過的に新形式へマイグレーションし、データの消失を防ぐ（あるいはパースエラーとみなして安全に初期化するフォールバック処理を行う）。
- **空白タスクの追加**: スペースのみの文字列、または空文字を入力してタスクを追加しようとした場合は追加処理を行わない（また、入力欄をハイライト・バリデーションエラーとして分かりやすくする）。
- **極端に長いタスク名**: アプリのレイアウトを破壊するような極端に長いタスク名（例: 数千文字）が入力された場合、適切に省略記号（...）で表示するか、複数行に適切に折り返して(word-break)表示し、UIが崩れないようにする。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001** (Must): ユーザーは新しいToDoアイテムをテキスト入力でリストの先頭（または末尾）に追加できること。
- **FR-002** (Must): ユーザーはToDoアイテムの完了チェックボックスを操作して、完了・未完了のステータスを即座に切り替えられること。
- **FR-003** (Must): ユーザーは既存のToDoアイテムのテキスト内容を編集して更新できること。
- **FR-004** (Must): ユーザーは任意のToDoアイテムを削除ボタンを用いてリストから削除できること。
- **FR-005** (Must): システムはタスクリストを「すべて」「未完了」「完了済み」の状態でフィルタリングし、該当するアイテムのみを表示できること。
- **FR-006** (Must): システムはタスクのデータ（一覧および状態）をデバイス内のLocalStorageに永続化し、次回のアプリ起動・リロード時に復元できること。
- **FR-007** (Must): システムはネットワークに接続されていないオフライン環境でも、全ての主要導線（追加、完了切替、編集、削除、閲覧、フィルタリング）が遅延なく動作すること。
- **FR-008** (Must): システムはローカルにある保存データが破損していたり不正な形式であったりする場合、パースエラーをハンドリングしてクラッシュを防ぎ、安全に初期状態のリストをロードすること。
- **FR-009** (Must)[a11y]: 全てのインタラクティブな要素（入力フィールド、追加ボタン、完了チェック、削除・編集ボタン、フィルター切替）は、マウスを使わずキーボード（Tabキー等）のみでアクセス・操作可能であること。
- **FR-010** (Must)[a11y]: 現在キーボードフォーカスが当たっている要素が、視覚的に明確（CSSのfocus-visibleなど）にハイライトされること。
- **FR-011** (Must)[a11y]: アイコンのみのボタンおよび全てのフォーム要素に対して、スクリーンリーダー向けの正しいラベル・aria属性が適切に設定されていること。

※スコープ外（Out of Scope）：
- ログイン機能、アカウント作成、クラウド同期、別ユーザーとのプロジェクト共有。
- プッシュ通知、期日や時間によるリマインダー通知の設定・発火機能。

### Key Entities

- **Task**: 
  - 属性: `id` (一意な識別子), `text` (タスク内容文字列), `completed` (完了状態のフラグ Boolean), `createdAt` (作成日時)。
- **Filter**:
  - `status`: 全て (All), 未完了 (Active), 完了 (Completed) のいずれかの状態を持つ。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが新規タスクの名前を入力し始めてから、リストに追加完了して表示されるまでの時間が【0.5秒未満】であること。
- **SC-002**: アプリが完全にオフラインの状態であっても、【初期表示が1秒未満】でロードされること（キャッシュなどの利用）。
- **SC-003**: タスクが1000件ほどローカルに保存されている状態でも、全ての操作（フィルター切り替えや追加・削除操作）のレスポンスが【カクつきなく、即座（0.15秒以内を目安）にUI反映】されること。
- **SC-004**: キーボードの「Tab」キーのみによる要素間の移動、「Enter/Space」キーによる実行のみで、タスクの新規追加から完了、テキスト編集モードの切り替え、リストのフィルタリング、削除までの全フローが完遂可能であること。
- **SC-005**: 開発者がブラウザのデベロッパーツール（Applicationタブ等）で手動データを意図的に「{ invalid json ]」のような不正フォーマットに上書きした後ページをリロードした際に、アプリ全体が白紙になる（クラッシュ）ことなくUIが表示されること。

## 用語定義

- **a11y**: Accessibility（アクセシビリティ）の略記。キーボード操作のみでの利用や、音声読み上げソフトなどを利用するユーザーにも使えるように配慮された設計要件。
- **ローカル永続化**: クラウドサーバーではなく、使っているコンピューターやスマートフォンのブラウザ内（LocalStorage）に直接データを保存しておく仕組み。
